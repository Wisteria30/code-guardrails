Hooksで「お願い」を「強制」に変えた

そこで入れたのが Claude Code Hooks です。

Claude CodeのHooksは PostToolUse でツール実行後に処理を挟めます。matcher に Edit|Write を指定すると、Claude がファイルを書いた直後だけフックできます。さらに decision: "block" を返すと、その場でClaudeを止めて修正を促せます。 ￼

設計として大事だったのは、「レビューの前に止める」ことでした。

レビューで見つけると、人間が読んで、指摘して、AIが直して、もう一回見る、という手戻りが発生します。

この差はかなり大きいです。

どういう設計にしたか

中身は、Rustでオーケストレーションを書いて、構文解析は ast-grep、広い範囲の候補探索は ripgrep に寄せています。READMEでもフルスキャン時は ripgrep で候補を絞ってから ast-grep を呼び、ルールの中核は rules/ 配下の ast-grep ルールで持つ構成になっています。 ￼

この分担にした理由は単純で、保存のたびに走るからです。
Hookのホットパスは遅いとすぐに嫌われます。
このリポジトリの公開 CLAUDE.md でも、scan-file / scan-hook は 100ms 以下を強く意識していて、役割分担も「rg は探索」「ast-grep は構文マッチ」「Rust は接着」に切っています。 ￼

今は Python と TypeScript のルールを持っていて、合計 17 ルールです。内訳は Python 9、TypeScript 8。テストパスや *.test.ts、**/tests/** のような場所は除外されています。 ￼

fail-open と fail-closed を分けた

ガードレールは強くしたい。
でも、強すぎて開発体験そのものを壊したくはない。

ここは設計でかなり気を使ったところです。

code-guardrails は、ポリシー違反を見つけたときは block します。

この分離は実務上かなり大事です。
「ルール違反だから止める」と「ツールが壊れたから全部止まる」は、まったく別の話だからです。

ガードレールは、コード品質のための仕組みであって、エディタを文鎮にする仕組みではない。
ここを混ぜると一気に嫌われます。

# 回答1
かなり本質的です。見えている問題は「検出精度」だけではなく、「違反したあとに AI が何を最適化するか」です。いまの code-guardrails は PostToolUse の Edit|Write をフックして違反時に decision: "block" を返す設計で、17 ルール・scan error は fail-open という構成です。ここまでは筋がいいです。けれど post-edit-scan.sh は scan 結果全体を additionalContext として Claude に渡していて、Claude Code 側ではこれは追加文脈として扱われます。するとモデルは「品質を上げる」より「フックを通す」を最適化しやすく、しかも主コンテキストを燃やしやすい。rename や構文変形で逃げるのは、ほぼ必然です。 ￼

なので、本当の解決は「fallback と test-double を禁止する」ではありません。「それらが発生したときの合法な解消先を、型・契約・アーキテクチャで先に用意する」です。乱暴に数理っぽく言うと、Option<T> -> T や Result<E, T> -> T には一般に正準な実装がありません。T に特権的な既定値が仕様で与えられたときだけ潰せます。つまり fallback は“親切な実装”ではなく、仕様が所有すべき写像です。ここが曖昧だと、AI もジュニアも必ず inline default に逃げます。

fallback の正解は、実務上は5種類しかありません。
1つ目は、合意済み default です。これはあなたの policy-approved 方向で合っていますが、コメント隣接よりさらに強いのは、許可済み default を専用 API に閉じ込めることです。approved_default("REQ-123", value, "ja-JP") や LocalePolicy.default_locale() のような blessed API だけを許し、生の .get(key, default) や ?? は全面禁止にする。コメントは人間に優しいですが、API は agent に優しいです。
2つ目は、境界での parse 不足です。Pydantic は untrusted data をモデルに通した結果が型制約に適合することを保証し、変換できなければ ValidationError を出します。BaseSettings も env や secrets から設定を集約できます。つまり os.getenv(..., default) をあちこちに散らすのではなく、入力・設定の境界で一度だけ parse/default し、その先の core では raw dict/env を触らないのが正解です。 ￼

3つ目は、本当に optional な値です。この場合は fallback で潰さず、Optional/union のまま型に残して、呼び出し側に exhaustive に処理させるべきです。TypeScript は strict と strictNullChecks で null/undefined を型として区別し、control-flow analysis で narrow でき、never で exhaustiveness を書けます。Python 側も mypy の --strict と assert_never() で union/Literal の取りこぼしを拾えます。 ￼
4つ目は、そもそも到達してはいけない状態です。ここは fallback ではなく契約違反です。Design by Contract の言葉で言えば、前提条件・事後条件・不変条件を破っているので、そこで止めて上位に伝播させるべきです。Eiffel の資料でも class invariant は valid/stable state を表し、check は「ここでは成り立っているはず」を明示するための仕組みです。Web ならその typed error を最外周で 4xx/5xx に写像すれば十分です。 ￼
5つ目は、resilience policy です。通信失敗時に cache や secondary に逃がすのは「その場しのぎ」ではなく、インフラ側の回復戦略です。これは adapter が所有し、監視・メトリクス・TTL・再試行方針まで含めて設計されるべきで、アプリケーションの呼び出し箇所が勝手に null や "default" を返してはいけません。

あなたの toolUseId の例は、まさに 2 か 4 です。正しい修正はこういう方向です。

from pydantic import BaseModel, ValidationError

class ToolUsePayload(BaseModel):
    toolUseId: str

def parse_tool_use(raw: dict) -> ToolUsePayload:
    try:
        return ToolUsePayload.model_validate(raw)
    except ValidationError as e:
        raise BadRequest("toolUseId is required") from e

tool_use = parse_tool_use(event.tool_use)
tool_use_id = tool_use.toolUseId

これで dict.get(..., "tool") も、if "toolUseId" in tool_use else "tool" も不要になります。直すべき場所は1行ではなく「境界」です。

test-double も同じです。本番の mock/stub/fake は名前の問題ではなく、依存グラフの問題です。runtime graph に test-support graph が混入しているのが本質です。なので原則はあなたの言うとおりで、「rename ではなく削除、必要なら一段上の設計修正」が正しいです。ただし実務ではひとつだけ大事な例外があります。本当に alternate implementation が必要なら、それはもう fake ではなく、明示的な runtime adapter に昇格させるべきです。SandboxPaymentGateway や NoopMailer や InMemoryEventStore は、意味が仕様化され、composition root で明示的に選ばれ、同じ port contract を満たすなら“正式な adapter”です。逆に、shape だけ同じ mock は観測可能な法則を保存する保証がないので、本番代替としては不適格です。各 adapter は同じ contract test を共有し、できれば stateful/property-based test で不変条件を確認するのがいいです。Hypothesis は操作列を生成し、各 step 後の invariants を検査できます。 ￼

高品質な全体設計に寄せるなら、私は次の6点に収束します。
境界で parse する。raw JSON / dict / env / queue payload をそのまま中へ入れない。
core は always-valid domain にする。entity や aggregate にグローバル default を持たせない。
default は policy object か settings module に閉じ込める。
error は explicit にする。domain/application では typed exception か Result、HTTP 写像は外周だけ。
adapter の選択は composition root だけがやる。
各 port には shared contract test を持たせ、型チェック・アーキテクチャテスト・property/stateful test を CI で回す。TypeScript は strict 系、Python は mypy/pyright の strict 寄せが効きます。 ￼

Claude/skill/hook の役割分担も変えた方がいいです。あなた自身の CLAUDE.md は hot path を 100ms 以下に保つことと、rg / ast-grep / Rust の責務分離を重視しています。なら hot path で「意味が同じ別構文」を無限に追うのは筋が悪いです。denylist を増やすより、合法な escape hatch をごく少数の allowlist に絞る方が強いです。加えて現状の TypeScript ルールは .ts/.cts/.mts 対象で .tsx/.jsx は未サポートなので、hook だけで完全性を狙うのも限界があります。 ￼

実運用では、Hook は「検出と短い分類」だけに寄せるのがいいです。additionalContext には全文レポートを流さず、violation_class=boundary-parse-missing のような短い分類と、詳細レポートのファイルパスだけ返す。重い是正は subagent に逃がす。Claude Code の docs でも、subagent は独立した context window で動き、高出力作業を main context から隔離するのに向いています。一方 Skills は main conversation context で使う再利用プロンプト向きです。つまり、skill には「解消 doctrine」を入れ、repair の実作業は subagent にやらせるのが自然です。subagent には skill を事前注入できます。ただし plugin subagent は hooks / mcpServers / permissionMode を持てないので、そこまで細かく制御したい repair agent は project/user scope に置く方が扱いやすいです。 ￼

AI やジュニアに教える doctrine は、結局これで十分です。

When a fallback/test-double violation fires, do not preserve the current line.
Repair at the owner layer.

1. Classify the owner:
   boundary / domain / infrastructure / composition root / test

2. Choose exactly one legal remedy:
   - approved policy default
   - boundary parser / settings model
   - Optional/union + exhaustive handling
   - typed exception / contract check
   - explicit resilience adapter
   - move double to tests
   - promote substitute to first-class adapter + contract tests

3. Forbidden moves:
   - rename mock/stub/fake
   - syntax-equivalent fallback rewrites
   - adding a new implicit default
   - keeping test-only dependencies in the runtime graph

4. Every fix must add one proof:
   - type exhaustiveness
   - contract/property/stateful test
   - architecture/import rule

要するに、fallback と test-double は line-level の悪癖ではなく ownership error です。
「この判断を誰が所有しているか」を先に決める。
spec が所有するなら policy。
境界が所有するなら parser。
到達不能なら exception。

定理3。fallback は「親切な実装」ではなく、型からは導けない追加構造です。

証明。
Maybe A を集合意味論で 1 + A と書きます。1 は「値がない」という 1 要素集合です。
fallback とは、これを A に潰す写像です。圏論の coproduct の普遍性より、

Hom(1 + A, A) ≅ Hom(1, A) × Hom(A, A)

です。もし「値があるときはその値を壊さない」という最低限の健全性を要求すれば、右側の第2成分は id_A に固定されます。すると必要なのは第1成分、すなわち

δ : 1 -> A

だけです。これは A の中の特別な一点、つまり default value の選択に他なりません。
同様に Result<E, A> = E + A なら

Hom(E + A, A) ≅ Hom(E, A) × Hom(A, A)

です。success を壊さないなら必要なのは h : E -> A、つまり「この error をどう value に回収するか」という handler です。

したがって、
	•	.get("timeout", 30) は δ(*) = 30 を選んでいる
	•	?? "demo" は δ(*) = "demo" を選んでいる
	•	.catch(() => null) は h(e) = null を選んでいる

のであって、どれも型だけからは出てきません。全部、仕様かポリシーかアーキテクチャが所有すべき追加意味です。
さらに強い事実として、Maybe => Id に対応する自然変換は存在しません。なぜなら A = ∅ と置くと 1 -> ∅ が必要になり、不可能だからです。Either_E => Id も、E ≠ ∅ なら E -> ∅ が必要になり同様に不可能です。
要するに、unapproved fallback とは「証明されていない追加構造を、行レベルで勝手に注入している」ことです。

系。
あなたの 4 分類はかなり正確で、数学的にはこう整理できます。absence / failure に遭遇したとき合法な道は本質的に四つしかありません。
第一に、仕様が所有する点 δ または handler h を与える。
第二に、sum を潰さず外へ伝播する。
第三に、そもそもその層に 1+A / E+A が現れないよう、手前で parse / validate して A にしてから渡す。
第四に、そこが到達不能だと契約で言えるなら exception / contract violation にする。
これ以外は全部「無許可の totalization」です。

補題。
|| は ?? より悪いです。?? は nullish だけを absence と見なしますが、|| は falsy を absence 扱いするので、0 や "" や false を「欠損」と同じ商類に潰します。つまり情報落ちがもっと荒い。|| が正当なのは、ドメインがそれらを本当に同一視しているときだけです。

定理4。production test-double 問題は、識別子問題ではなく model theory の問題です。

証明。
port を「メソッド名と型」だけではなく、「守るべき法則」を含む理論 T = (Σ, Ax) とみなします。
ここで Σ はシグネチャ、Ax は法則です。たとえば repository なら、
	•	save(u); get(id(u)) = Some(u)
	•	delete(id); get(id) = None
	•	必要なら整合性・順序・可観測性

のような law が Ax に入ります。
型検査や nominal/structural interface が示すのは、せいぜい「この実装は Σ を満たす」ということです。しかし Ax は別です。Σ を満たすが Ax を満たさない実装は簡単に作れます。hard-coded に決まった値を返す repository も、型だけ見れば repository です。
したがって、mock を secondly に rename しても、理論のモデルであるかどうかは1ミリも改善しません。変わるのは lexical signal だけです。

ここで契約の観点を入れると、合法な代替実装とは「同じ contract を満たす substitute」です。Eiffel の Design by Contract は、precondition / postcondition / invariant によって routine/class の契約を明示し、継承では precondition は弱く、postcondition は強くなければならないと述べています。つまり合法な代替は、クライアントに要求を増やさず、約束を減らさず、不変条件を破らないものです。これが behavioral substitutability の核です。  ￼

系。
本番で許される “fake” は、fake ではなく first-class adapter に昇格したものだけです。SandboxPaymentGateway、NoopMailer、InMemoryEventStore が合法になりうるのは、名前が穏当だからではなく、同じ port theory のモデルとして contract suite を通るからです。逆にそこまで証明できないものは、rename ではなく runtime graph から削除が正しい。

定理5。型が弱いと fallback は「悪い癖」ではなく「合理的な自己防衛」として発生します。

証明。
関数の意味が本来 V ⊂ X という valid subset 上でしか定義されていないのに、実装者に X 全体が見えているとします。すると total function を書くには三択しかありません。
	•	X を V へ縮める。つまり parse / validate / refine する。
	•	X \ V を explicit な error / option / union として返す。
	•	X \ V 上で恣意的な点を選ぶ。これが fallback。

つまり fallback は、型・境界・契約が不足したときの数学的帰結です。
TypeScript では strict が強い型検査群を有効にし、strictNullChecks は null / undefined を distinct type にします。さらに exhaustive handling は never ベースで書けます。Python では逆に、標準 typing は runtime ではアノテーションを強制しません。だから Python では static type だけでは足りず、境界で runtime validation を入れる必要があります。Pydantic の BaseModel / model_validate はそのための道具で、extra='forbid' なら余分な入力に対して ValidationError を上げられます。Python 側の exhaustive check には typing.assert_never() が使えます。  ￼

だからあなたの toolUseId の例の正解は、行の書き換えではなく境界の精製です。

from pydantic import BaseModel, ConfigDict

class ToolUsePayload(BaseModel):
    model_config = ConfigDict(extra='forbid')
    toolUseId: str

payload = ToolUsePayload.model_validate(event.tool_use)
tool_use_id = payload.toolUseId

これで core 側では tool_use_id: str が total value になり、fallback 自体が不要になります。
ここで重要なのは、「fallback を禁止した」のではなく、「その層で fallback を表現する必要を消した」ことです。

定理6。Edit|Write ごとの hard block は、prefix-safe でない性質に対しては正しい大域修理を妨げます。

証明。
編集を遷移 p -> p' とするグラフを考えます。hook が毎 edit 後に A ⊂ P を強制するとは、「到達可能な経路は常に A の中に留まる」ということです。
もし望ましい最終状態 p* ∈ A に至るすべての経路が、一時的に A の外へ出るなら、p* は到達不能です。QED。

これは非常に重要です。
secret 漏えい、危険な shell、権限逸脱のような irreversible harm は prefix-safe に block すべきです。
しかし fallback/test-double のような architectural smell は、multi-file refactor や owner-layer への移送を要することが多く、最終状態で検査する方が自然なことがあります。Claude Code には Stop / SubagentStop で「そこで終わるのを止める」フックもあります。つまり「各 edit で即 block」しか選べないわけではありません。今の plugin は PostToolUse の Edit|Write に hard block を置いていますが、これは prefix-safe でない品質性質まで prefix-safe に扱っている、ということです。  ￼

即時 hard block は、prefix-safe で局所修復可能なものだけ。

定理7。解くべきは「bad pattern detection」ではなく「proof-carrying repair」への還元です。

fallback が正当か? や この substitute は本番で合法か? を checker が推論しようとすると、意味論・文脈・仕様・将来の責務分担まで読む必要があります。これは hot path の deterministic hook には重すぎますし、あなたの repo 自体も hot path は 100ms 以下、rg / ast-grep / Rust glue の役割分担を強調しています。Claude Code 側でも hooks は loop 外の deterministic script です。

ただしコメントは弱い witness です。人は書けるし AI も書ける。真に強い witness は、型・API・registry・contract suite に昇格したものです。たとえば

のように、合法な totalization / substitution のみが特定の構文からしか生成されないようにする。

ここから導かれる全体設計は、かなりはっきりしています。

第一原理1。absence / failure は effect であり、value ではない。
したがって .get(default)、??、catch { return ... } は “便利記法” ではなく effect handler です。handler は owner layer でしか書いてはいけません。

第一原理2。owner layer は有限個しかない。
境界、ドメイン、アプリケーション、インフラ、composition root、テストです。
それぞれの責務はこうです。

境界は parse / validate / normalize を所有します。raw dict、raw JSON、raw env、raw queue payload はここで止める。Python では runtime type enforcement がないので、Pydantic などの runtime validator を境界に置くのが本質です。  ￼

ドメインは invariant を所有します。always-valid domain にする。コンストラクタと exported routine が invariant を確立・維持すれば、クライアントは invalid state を観測しません。fallback で “なんとかする” 必要が減るのはここです。Eiffel の invariant / pre / post はこの思想を最も明示的に書いています。  ￼

アプリケーション層は orchestration を所有します。境界から受け取った explicit error / union を、どこで translation するかを決める。ここで勝手に default しない。必要なら explicit exception / typed error を上へ運ぶ。

インフラ層は resilience policy を所有します。retry、cache、secondary、degrade mode はここで初めて合法です。ただし handler E -> A はメトリクス、TTL、整合性条件、可観測性を含む policy でなければならない。単なる .catch(() => null) ではありません。

composition root は implementation choice を所有します。代替実装の選択はここでだけ行う。port theory のモデルとして承認された adapter しか注入してはいけない。mid-flow に FakeRepository() を置くのは ownership violation です。

テストは double を所有します。ここだけは test oracle の都合で partial model を使っていい。しかし production に持ち込むなら、それはもはや test double ではなく first-class adapter です。

この設計から、AI/junior に与えるべき判定手順も一意に出ます。
	1.	いま書こうとしているのは 1+A -> A か E+A -> A の消去か。
	2.	それを所有するのはどの層か。
	3.	そこに合法な witness はあるか。
	4.	なければ、default を書くな。sum を運ぶか、境界を精製するか、契約違反として止めろ。
	5.	いま入れようとしている代替実装は port theory の model か。
	6.	証明できないなら rename するな。runtime graph から消せ。

これを tool に落とすなら、私はこう設計します。

hook は verifier に徹する。
hot path では semantic diagnosis をしない。やるのは「違反クラスの分類」「witness の有無の確認」「軽量な owner 推定」だけ。scan 詳細全文を additionalContext に流すのはやめ、短い violation code と詳細レポートへのファイルパスだけを渡す。hooks は deterministic script、しかもあなたの repo は hot path latency を強く意識しているので、ここで意味論探索をやるべきではありません。  ￼

skill は doctrine に徹する。
skill には「owner を探す手順」「合法 remedy の列挙」「禁止される逃げ方」「必要な witness」を書く。skills は reusable knowledge / workflow で、読み込まれた skill 自体は main context に入ります。だから、ここは playbook として短く強く保つ。  ￼

subagent は repair に徹する。
reasoning-heavy な修理、複数ファイル横断の owner-layer refactor、contract test 追加は subagent にやらせる。subagent は独立 context window で動いて summary を返すので、main context を汚しにくい。fallback/test-double 問題はまさにこちら向きです。  ￼

検査軸も変えるべきです。
“mock という文字列があるか” ではなく、
“runtime graph が test-support に依存していないか”。
“.get(default) があるか” ではなく、
“unauthorized coproduct eliminator が owner layer 以外にないか”。
“例外を握りつぶしていないか” ではなく、
“effect handler が policy / adapter / transport boundary に局在しているか”。
この軸なら rename や構文同値変形に強くなります。

最後に、law の証明方法です。
adapter の合法性は example-based test だけだと弱いです。性質として書くべきです。property-based testing は「任意入力で保たれるべき law」を書けますし、Hypothesis の stateful testing は値だけでなく操作列全体を生成して、stateful adapter の invariant 破れを探せます。repository、cache、event store、workflow engine のような stateful port にはとても相性がいいです。  ￼

結論を一行で言います。
本当の不変条件は、「mock や ?? を禁止する」ではありません。
「すべての coproduct/effect の消去と、すべての runtime implementation の置換に、正しい owner と machine-checkable witness が付いていること」です。

これを満たすように設計すると、
	•	fallback は境界・政策・resilience に吸収される
	•	test-double は tests へ閉じるか、lawful adapter へ昇格する
	•	AI は rename で逃げても意味がなくなる
	•	hook は軽く、subagent は深く、main context は汚れにくくなる

つまり、fallback と test-double は「悪い書き方」の問題ではなく、
無所有の消去 と 無証明の置換 の問題です。
そこまで抽象化すると、解決策は禁止ではなく、証明可能な構成へ押し戻すことだと分かります。
