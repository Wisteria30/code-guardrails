アウフヘーベンすると、次版の code-guardrails は「fallback と test-double を見つけて止める plugin」ではなく、「無所有の消去と無証明の置換を、正しい owner と witness を持つ形へ押し戻す repair system」にするのが筋です。回答1の owner-layer 論と、回答2の Option/Result 消去・lawful substitute の形式化は、実は同じ一点を見ています。問題は検出漏れそのものより、違反後の AI が“品質”ではなく“hook 通過”を最適化してしまうことです。 ￼

現行 repo は README どおり PostToolUse で Edit/Write 後に changed file をスキャンし、17 ルールを Rust + ast-grep + ripgrep で回しています。実装側では detect_rule_ids() が lowercased content に対する contains() を前段選別にかなり使っており、hook script は scan 結果全文を additionalContext に入れて decision: "block" を返しています。Claude Code docs でも additionalContext は Claude の文脈に追加されるので、この構成だとモデルは「owner layer まで戻って直す」より「少し表現を変えて今の block を外す」方に寄りやすいです。/scan も違反列挙後に fix か policy-approved コメント追加を促す流れなので、repair が line-level patch に寄りやすいのは自然です。 ￼

だから次版で保存すべきなのは、今の fast path と fail-open の哲学です。scanner 自体の故障で開発体験を壊さないこと、rg/ast-grep/Rust の責務分離、hot path を短く保つことはそのまま残すべきです。Anthropic docs でも hooks は deterministic script として agent loop の外に置かれていますし、repo の CLAUDE.md も scan-file hot path を 100ms 以下で保つことを強く意識しています。重い意味論探索を sync hook に押し込むのは逆方向です。 ￼

ここで理論を一段だけ上げると、fallback は「親切な書き方」ではありません。.get(k, d) や ?? d は 1 + A -> A の一点選択で、.catch(() => v) は E + A -> A の handler 選択です。型だけでは決まらない追加構造を、その場で勝手に注入しています。つまり unapproved fallback の本質は「AI が仕様の owner を代行している」ことです。

test-double も同じで、本質は名前ではありません。Mock を secondly に rename しても、runtime graph に test-only semantics が混入している事実は変わりません。型や interface の shape が合っても、port の laws を満たす lawful model かどうかは別問題です。だから production で許される代替実装は、test double ではなく first-class adapter に昇格したものだけです。

なので、次版の不変条件は一つです。

すべての不確実性の消去と、すべての runtime 実装の置換に、正しい owner と machine-checkable witness を要求する。

これを実務に落とすと、owner は 6 つだけです。
	1.	boundary
raw JSON / dict / env / queue payload を parse・validate・normalize する場所。
	2.	domain
always-valid domain と invariant を保持する場所。
	3.	application
use case の分岐と typed error の伝播を担う場所。
	4.	infrastructure
retry / cache / secondary / degrade mode など resilience policy を持つ場所。
	5.	composition root
どの adapter 実装を使うかを選ぶ場所。
	6.	tests
double を持ってよい唯一の場所。

boundary だけは tool でもかなり強く押せます。Python runtime は型注釈を実行時に強制しないので、raw dict や TypedDict を core にそのまま流すと、fallback が合理的な自己防衛として発生します。だから inbound data は Pydantic model_validate() などで一度 model 化し、必要なら extra='forbid' で未知キーも拒否する。TypeScript 側は strictNullChecks で null/undefined を explicit にし、never で exhaustiveness を取る。Python でも typing.assert_never() で同じ発想を持ち込めます。 ￼

あなたの toolUseId 例の正解は、まさに line repair ではなく boundary repair です。

from pydantic import BaseModel, ConfigDict, ValidationError

class ToolUsePayload(BaseModel):
    model_config = ConfigDict(extra='forbid')
    toolUseId: str

def parse_tool_use(raw: dict) -> ToolUsePayload:
    try:
        return ToolUsePayload.model_validate(raw)
    except ValidationError as e:
        raise BadRequest("toolUseId is required") from e

payload = parse_tool_use(event.tool_use)
tool_use_id = payload.toolUseId

この形にすれば、.get("toolUseId", "tool") も、if "toolUseId" in tool_use else "tool" も、どちらも不要になります。fallback を禁止したのではなく、その層で fallback が必要になる理由を消しています。 ￼

test-double 側の正解も二択しかありません。tests に閉じ込めるか、runtime adapter に昇格させるかです。FakeUserRepository を production に残して rename するのは最悪で、正しくは「削除して依存注入を設計し直す」か、「SandboxUserRepository や InMemoryEventStore のような正式 adapter に改名し、composition root でのみ選び、shared contract suite を通す」です。状態を持つ port なら example-based test だけでは弱いので、property-based / stateful test を contract suite に足すのがよいです。Hypothesis は値だけでなく action sequence も生成できます。 ￼

ここから導かれる vNext の doctrine はかなりシンプルです。AI とジュニアに教えるべきことは、禁止構文の一覧ではなく、次の意思決定です。 ￼
	1.	いまやろうとしているのは、Option<T> -> T / Result<E, T> -> T の消去か、runtime substitute の導入か。
	2.	それを所有するのはどの layer か。
	3.	許される remedy は 1 つだけ選ぶ。
	4.	remedy に対応する witness を 1 つ残す。
	5.	rename と同値書き換えは禁止。

許される remedy は 7 つで十分です。
	•	approved policy default
	•	boundary parser / settings model
	•	Optional / union のまま保持して exhaustive handling
	•	typed exception / contract violation
	•	explicit resilience adapter
	•	move double to tests
	•	promote substitute to first-class adapter + contract tests

禁止手は 5 つです。
	•	mock/stub/fake の rename
	•	.get(default) を if key in dict else default に変えるだけの同値回避
	•	新しい inline default の追加
	•	新しい policy-approved コメントの捏造
	•	runtime graph への test-support 混入

そして every fix に proof を 1 つ要求します。
	•	parser/schema による boundary proof
	•	strictNullChecks / never / assert_never() による exhaustiveness proof
	•	contract/property/stateful test
	•	architecture/import rule
	•	registry-backed approval id

これをそのまま code-guardrails vNext の具体設計に落とすと、構成は 3 層です。

verifier は今の Rust + rg + ast-grep を維持します。仕事は「検出・分類・レポート保存」まで。
doctrine は skill に移します。仕事は「owner を決める手順と legal remedy の playbook」を与えること。
repairer は subagent に移します。仕事は「複数ファイルを跨いだ owner-layer refactor と proof 追加」です。

この分割は Claude Code の extension model と相性がいいです。skills は on-demand で読み込め、disable-model-invocation: true にすれば説明自体を通常 context に載せず、context: fork + agent で isolated context 実行もできます。subagents は main conversation と別 window で動いて summary だけ返します。plugin は skills/, agents/, hooks/ をまとめて配布できます。現行 repo の tree には top-level agents/ が見当たらないので、次版ではここを足すのが効きます。custom commands は skills に統合済みなので、今の commands/scan.md も新規設計では skill 化したほうが素直です。 ￼

plugin 構成は、私はこうします。

  skills/

guardrail-repairer は owner-layer 修理専用 agent、repair-guardrail はそれを forked context で呼ぶ manual skill、scan-project は現行 /scan を置き換える skill です。plugin agents も使えますが、plugin-shipped agents には hooks / mcpServers / permissionMode を持たせられないので、repair agent にそこまでの制御が要るなら project/user scope に逃がす設計にします。 ￼

hook flow も分けます。
	•	PostToolUse sync command hook
変更ファイルを scan して violation を分類し、詳細 JSON を ${CLAUDE_PLUGIN_DATA} 以下に保存する。Claude には全文を食わせず、短い violation capsule だけ返す。
	•	PostToolUse async command hook
背景で scan-tree、contract smoke、必要なら changed area の type/test を走らせる。async hook は block できないので、これは audit 専用に割り切る。
	•	Stop agent hook
authoritative gate。未解決 guardrail report が残っていたら { "ok": false, "reason": ... } を返し、Claude を継続させる。architectural smell を “各 edit で完全に clean であること” ではなく “最終停止時に unresolved でないこと” に変える。

Claude docs では PostToolUse と Stop の両方で prompt/agent hooks が使え、async は command hook のみ、async は block 不可、Stop hook は ok:false で Claude を継続させられます。これを使うと、prefix-safe でない品質性質を無理に every-edit hard block しなくて済みます。 ￼

ここで最重要なのは、hook の出力チャネルを変えることです。現行は scan 結果全文を additionalContext に流していますが、vNext では詳細はレポートファイルへ、Claude には最小限の capsule だけにします。docs 上も systemMessage は user 向け警告、additionalContext は Claude 向け文脈です。だから user には report path と件数、Claude には owner guess・legal remedies・forbidden moves だけ渡すのがよいです。 ￼

たとえば返す JSON はこのくらいで十分です。

{
  "decision": "block",
  "reason": "Guardrail: fallback_unowned_default at src/events.py:42. Repair at boundary or raise typed error.",
  "systemMessage": "Detailed report saved to ${CLAUDE_PLUGIN_DATA}/reports/cg-20260329-abc123.json",
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "guardrail_id=cg-20260329-abc123 class=fallback_unowned_default owner_guess=boundary remedies=boundary_parser|typed_exception|approved_policy_default forbidden=rename|equivalent_rewrite|new_inline_default"
  }
}

この形なら main context を焼かずに済みますし、Claude にも「何をしてはいけないか」ではなく「どこで直すべきか」が伝わります。

approval model も上げます。現行 README の adjacent policy-approved コメントは残してよいですが、次版では migration escape hatch に格下げです。happy path は comment ではなく registry-backed blessed API にします。さらに AI は新しい approval id を勝手に生やせないようにします。ID は policy/defaults.yml に事前登録されていなければ無効、adjacent comment はその登録済み ID を参照するだけ、がよいです。 ￼

しかも generic helper approved_default(req, value, fallback) を最終形にしないほうがいいです。これは comment よりは強いですが、まだ意味が helper に逃げています。bounded context に責務を凝縮するなら、次版は LocalePolicy.default_locale(raw), DemoLabelPolicy.resolve(apiValue), PricingPolicy.default_currency(raw) のような context-specific policy API を blessed symbol にするのがよいです。registry はその symbol と REQ/ADR を結ぶだけにする。これで AI は「generic fallback helper を呼べばよい」と学ばず、「default は policy object の責務」と学べます。

rule 側の分類も、syntax 名から semantics 名に寄せます。次版で増やすのは endless denylist ではなく、この 6 クラスです。
	•	fallback_unowned_default
	•	fallback_unowned_handler
	•	boundary_parse_missing
	•	optional_not_exhaustive
	•	runtime_double_in_graph
	•	adapter_choice_outside_composition_root

今の .get(default) / ?? / catch / mock|stub|fake ルールは、これらの class にぶら下がる surface detector にします。こうすると「同じ意味の別構文」を追いかけるのではなく、「結局どの semantics class に属するか」を中心に修理 doctrine を組めます。

さらに強くするなら、denylist ではなく choke point を締めます。たとえば、
	•	env access は settings module 以外で禁止
	•	inbound payload の parse 前アクセスは boundary 以外で禁止
	•	runtime から tests/, fixtures/, unittest.mock への import を禁止
	•	adapter instantiation は composition root 以外で禁止
	•	inline catch-return-default は infrastructure の approved resilience module 以外で禁止

こうすると、dict.get(..., d) を if "k" in d else d に書き換えても逃げにくくなります。syntax ではなく ownership violation を取るからです。

CI 側の証明も重要です。高品質なソフトウェアになるかどうかは、hook 単体では決まりません。最低でも 4 本です。
	•	strict type checking
TS は strictNullChecks と never exhaustiveness。Python は strict な type checker と assert_never()。
	•	boundary validation
Python は Pydantic などで inbound data を model 化。未知キーは extra='forbid' を基本線にする。
	•	architecture tests
layer 越境 import、test-support runtime import、composition root 逸脱を CI で落とす。
	•	contract/property/stateful tests
approved runtime adapters は shared contract suite を通す。stateful port は Hypothesis stateful まで入れる。

これは Design by Contract の方向でもあり、仕様と invariant を実行可能な形に近づける設計です。 ￼

そのまま貼るなら、AI/junior 向け doctrine はこれで十分です。

## Guardrail Repair Doctrine

A fallback is an effect handler, not a convenience.
A production substitute is an adapter, not a fake.

When a guardrail fires, never preserve the violating line.

1. Find the owner layer:
   boundary / domain / application / infrastructure / composition root / test

2. Choose exactly one legal remedy:
   - boundary parser / settings model
   - Optional/union + exhaustive handling
   - typed exception / contract violation
   - approved policy API
   - explicit resilience adapter
   - move double to tests
   - promote substitute to first-class adapter + contract tests

3. Add one proof:
   - schema/parser
   - exhaustiveness check
   - architecture rule
   - contract/property/stateful test
   - registered approval id

Forbidden:
- renaming mock/stub/fake
- syntax-equivalent fallback rewrites
- adding a new inline default
- inventing a new approval id
- importing test support into runtime

CLAUDE.md に置くのはこの短い doctrine だけにして、長い例と手順は skill に逃がすのがよいです。Claude docs でも CLAUDE.md は毎 session load され、長くなるほど context を食うので、長い reference は skills や rules に移す方がよいとされています。 ￼

実装順もはっきりしています。
	1.	まず出力チャネルを変える。
full findings を additionalContext に流すのをやめる。これだけで main context burn はかなり改善します。
	2.	次に repair skill / repair agent を入れる。
repair-guardrail を context: fork, disable-model-invocation: true で作る。深い修理は isolated context に逃がす。
	3.	そのあと Stop gate を入れる。
every-edit clean ではなく final unresolved = NG にする。
	4.	次に policy registry を入れる。
comment-only approval をやめ、registered id + blessed symbol へ移行する。
	5.	最後に architecture / contract proof を足す。
ここまで来てから rule surface を広げる。今は .tsx/.jsx 対応より owner/witness モデルの導入が先です。現行 repo 自身も TypeScript rule 対象を .ts/.cts/.mts に絞っています。 ￼

一行で締めると、次版の不変条件は「mock や ?? を消す」ではありません。

不確実性の消去と実装の置換が、正しい owner で、正しい witness 付きでしか起きないこと。

ここまで上げると、AI は“うまく誤魔化す”のではなく、“どの層でどう直すか”を学びます。guardrail が禁止装置から、設計を良い方向へ折り返す装置に変わります。
